## What Changed

### 1) Prevent lost Telegram updates on enqueue failure
- If webhook processing marks an update as processed but enqueue to Redis fails, the processed marker is now rolled back.
- This avoids a permanent silent drop of the Telegram update.

Files:
- `api/src/app.ts`
- `api/src/server.ts`
- `shared/src/updates.ts`
- `api/src/selftest.ts`

### 2) Add hard timeouts for Telegram API calls
- Added explicit abort-based timeouts for:
  - `sendMessage`
  - `sendChatAction`
  - `setWebhook`
  - `getFile`
  - Telegram file downloads
- This prevents worker slots from hanging indefinitely on external Telegram requests.

Files:
- `shared/src/telegram.ts`

### 3) Reduce worker concurrency to avoid race conditions
- Telegram worker concurrency is now configurable via `TELEGRAM_WORKER_CONCURRENCY`.
- Default changed to `1`.
- This is safer for the current expected load (up to ~2 users) and reduces out-of-order processing and LLM contention.

Files:
- `shared/src/config.ts`
- `shared/src/queue.ts`
- `workers/src/index.ts`
- `.env.example`
- `.env` (local only)

### 4) Prevent duplicate replies after successful Telegram send
- After a Telegram message is already delivered, any subsequent DB persistence failure no longer fails the whole job.
- Post-delivery persistence errors are logged and skipped instead of retried.
- This avoids sending the same user-facing reply twice on BullMQ retry.

Files:
- `workers/src/index.ts`

### 5) Add fail-fast Redis startup checks
- `api` and `worker` now check Redis availability during bootstrap before continuing.
- Startup now exits immediately with a clear fatal error if Redis is unavailable.
- Error formatting was improved to show useful root causes such as `ECONNREFUSED host:port`.

Files:
- `shared/src/redis.ts`
- `api/src/index.ts`
- `api/src/server.ts`
- `workers/src/index.ts`
